# Electron Desktop Packaging and Semi-Auto Update Plan

## 1. 决策摘要

- 桌面端技术路线选择为 Electron。
- 保持单一业务代码源：复用现有 `@nextclaw/core`、`@nextclaw/server`、`@nextclaw/ui`，不维护两套前端。
- 桌面用户安装不依赖系统 Node/npm；通过安装包即可运行。
- 更新策略采用半自动更新：应用内检查、下载、提示重启安装，同时保留手动下载兜底。

## 2. 背景与目标

- 目标 1：最低长期维护成本，不拆分业务实现。
- 目标 2：普通用户可直接安装使用，不要求先装 Node。
- 目标 3：版本升级体验接近原生桌面应用，避免每次都让用户手工重新查找安装包。
- 目标 4：沿用现有发布体系，尽量减少架构震荡和迁移风险。

## 3. 非目标

- 不在第一阶段引入全自动静默更新。
- 不在第一阶段改造为 Tauri 或引入 Rust 工具链。
- 不在第一阶段重写核心服务逻辑或替换现有 API 协议。

## 4. 总体架构

- `apps/desktop`：Electron 壳层（main、preload、打包配置）。
- `packages/nextclaw-ui`：继续作为 renderer 前端。
- `packages/nextclaw` + `packages/nextclaw-server` + `packages/nextclaw-core`：继续作为本地服务运行时。
- 桌面壳负责进程编排、窗口、托盘、更新，不承载业务逻辑。

## 5. 启动与退出模型

- 启动流程：
  1. Electron `main` 启动后申请单实例锁。
  2. 选择可用 UI 端口并拉起本地服务进程。
  3. 轮询健康检查（如 `/api/config`）通过后加载窗口 URL。
  4. 健康检查失败时展示错误页（重试、查看日志、复制诊断信息）。
- 退出流程：
  1. 先发送优雅停止信号。
  2. 超时后强制结束子进程。
  3. 清理锁文件和临时状态，避免端口残留。

## 6. 安装与打包策略（无 Node 预装前提）

- 发布物：
  - macOS：`dmg`（可补 `zip` 供更新通道使用）。
  - Windows：`nsis` 安装包。
- 用户侧要求：
  - 不要求预装 Node/npm。
  - 下载安装包后可直接启动桌面应用。
- 构建原则：
  - Electron 包内携带运行 NextClaw 所需资产。
  - Electron 主进程负责拉起服务，不依赖用户本机全局 npm install。
  - 开发者渠道保留 `npm install nextclaw`，但不作为桌面版主路径。

## 7. 半自动更新方案

- 更新体验：
  1. 应用启动后和固定周期检查新版本。
  2. 发现新版本时自动下载更新包到本地缓存。
  3. 下载完成后提示用户“重启并安装”。
  4. 用户确认后退出并应用更新。
- 失败兜底：
  - 任一步失败时降级到手动下载页，保证可恢复。
- 通道策略：
  - 默认 `stable` 通道。
  - 可选 `beta` 通道（内部/灰度用户）。
- 关键要求：
  - macOS/Windows 均需代码签名；macOS 需要 notarization。
  - 发布资产与更新元数据必须同版本闭环上传。

## 8. 版本与发布策略

- 建议维护两个语义版本：
  - 桌面壳版本（Electron App）。
  - 内置运行时版本（NextClaw runtime）。
- 发布时记录版本映射，避免“壳版本升级但运行时未对齐”。
- 升级兼容原则：
  - 小版本优先做平滑升级。
  - 破坏性变更通过 release notes 和应用内提示明确告知。

## 9. 安全基线

- Electron 安全配置：
  - `contextIsolation: true`
  - `nodeIntegration: false`
  - 通过 preload 白名单暴露 IPC
- 本地服务访问保护：
  - 限制允许来源。
  - 增加会话级 token 或等价防护，避免本机其他页面滥用本地接口。
- 日志与诊断：
  - 桌面端日志落盘。
  - 错误页可导出诊断信息，便于支持排障。

## 10. 交付阶段

- 阶段 A（最小闭环）：
  - 可安装、可启动、可关闭、可手动更新兜底。
- 阶段 B（半自动更新）：
  - 应用内检查与下载更新、重启安装、失败回退。
- 阶段 C（发布完善）：
  - 代码签名、notarization、稳定发布流程与回滚手册。

## 11. 验收标准

- 安装验收：
  - 在未安装 Node 的机器上，用户可安装并进入主界面。
- 运行验收：
  - 应用退出后无残留服务僵尸进程。
- 更新验收：
  - 新版本可被检测、下载并在重启后生效。
  - 更新失败时可一键跳转手动下载，不阻断当前版本继续使用。

## 12. 执行建议

- 先落地“可运行 + 可升级”主链路，再逐步接入托盘、开机启动、深链等增强能力。
- 保持平台差异封装在 desktop adapter，避免业务逻辑回流到 UI 页面层。
- 每次发布前固定执行安装与更新冒烟，形成标准化发布检查单。
