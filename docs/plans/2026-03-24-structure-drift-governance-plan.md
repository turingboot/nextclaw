# Nextbot Structure Drift Governance Plan

**Goal:** 把 `nextbot` 的治理视角从“只在文件过长或函数过复杂时报警”升级为“能识别模块结构漂移、默认阻断新增结构债务、对高风险目录建立红区冻结和拆分缝闭环”的长期机制。

**Why Now:** `workers/nextclaw-provider-gateway-api/src` 这次暴露的问题，不只是若干文件放错目录，而是同一模块同时混用了多种组织方式，说明仓库已经出现“结构熵”增长。继续只靠文件级 maintainability 规则，只能看到症状，拦不住同类问题在别的模块重复出现。

## 1. 这份方案要解决什么

这份方案不是为了要求“所有目录立刻重构完”，也不是为了把所有项目都改成一种绝对统一的目录结构。它要解决的是另一层问题：

- 如何把一个局部坏样本当成镜子，识别更深的机制缺口
- 如何把“这个目录越来越难维护”的感觉，翻译成稳定、可重复、可验证的治理信号
- 如何像现有 maintainability skill 一样，把结构问题从“靠经验提醒”升级为“有默认守卫、有红区、有留痕、有下一步拆分缝”

换句话说，本方案治理的对象不是单个文件，而是模块结构如何随时间漂移。

---

## 2. 从镜子样本看到的系统性问题

`workers/nextclaw-provider-gateway-api/src` 之所以有代表性，不在于它有 `utils` 或 `types` 目录，而在于它暴露了这些系统性缺口：

### 2.1 我们有文件级治理，但缺少模块级治理

当前仓库已经能抓到：

- 超长文件
- 超长函数
- 命名职责错配
- 红区文件继续膨胀

但还抓不到：

- 同一职责出现两套平行入口
- 同一模块混用两到三种目录组织模型
- 共享目录逐步退化成垃圾桶
- 新结构替代旧结构后，旧路径仍长期存活

这说明我们缺的是“结构漂移治理层”。

### 2.2 我们缺少模块拓扑契约

很多目录之所以越来越乱，不是因为某个人不会起名字，而是因为从一开始就没有写清：

- 这个模块是 `feature-first`，还是 `layer-first`
- 根目录允许放什么，不允许放什么
- `utils`、`types`、`support`、`common` 这些共享容器的适用边界是什么
- 一个新结构替代旧结构后，旧结构应何时退役

没有拓扑契约，结构就会在新增功能时自然漂移。

### 2.3 我们缺少“替代后退役”机制

很多结构漂移不是一次大事故，而是下面这种温和但持续的过程：

1. 新增一份更合理的实现
2. 旧实现先留着“以防万一”
3. 新旧入口并存数周甚至数月
4. 后续维护者不知道哪一个才是事实来源

这类问题不会被文件长度规则抓到，但对维护性伤害极大。

### 2.4 我们缺少目录级红区治理

当前红区机制主要面向单文件。但结构问题往往不是一个文件失控，而是整个目录的组织方式开始失真：

- 根目录业务文件持续增多
- 同一 feature 横跨多个技术层目录才能拼全
- shared 目录不断吸纳跨域内容

这类问题需要目录级别的热点冻结，而不是只盯单文件。

---

## 3. 结构漂移的定义

本方案把“结构漂移”定义为：模块原本可预测、可解释、可演进的目录和职责边界，随着迭代逐渐变得混杂、模糊、并行、不可预测的过程。

以下现象都属于结构漂移信号：

- 同一职责出现多份主入口或多份注册器
- 一个模块同时混用 `feature-first`、`layer-first`、根目录散文件三种组织方式
- 根目录逐步吸纳本应属于 feature 的业务文件
- `utils`、`types`、`support`、`common`、`helpers` 之类目录/文件不断吸纳跨域逻辑
- 单一业务域内容被拆散在多个宽泛目录中，导致维护者必须靠记忆找文件
- 新文件名看似合法，但其实际路径让模块边界更模糊
- 新结构已出现，但旧结构没有退役计划

结构漂移的重点不是“目录名好不好看”，而是“维护者是否还能预测改动应该落到哪里，以及系统是否只有一个明确的事实来源”。

---

## 4. 治理目标

### 4.1 不再只做事后抱怨

当我们说“这个目录越来越难维护”时，系统应该能给出可验证信号，而不是停留在主观感受。

### 4.2 先阻断新增漂移，再渐进收敛存量

本方案不要求一次性把历史目录都重构干净，而要求：

- 新增结构债务默认更难进入主干
- 存量坏样本被标成红区并冻结
- 每次触达红区时都留下下一步拆分缝

### 4.3 建立模块级单一事实来源

对于路由入口、主装配器、runtime 入口、feature 主注册点等核心结构，必须尽量只保留一个主路径。

### 4.4 共享目录可以存在，但必须有边界

本方案不追求消灭 `utils/` 或 `types/`，而是要求它们只承担真正共享、稳定、可预测的职责，不能成为 feature 边界失效后的兜底垃圾桶。

---

## 5. 治理原则

### 5.1 先看机制缺口，再看局部目录

每次发现目录难维护时，先问：

- 这是个例，还是机制没覆盖？
- 能否把这类判断沉淀成 guard/脚本/红区规则？
- 以后如何让同类问题默认更难出现？

除非明确是局部偶发问题，否则不接受“只修这个目录，不补机制”。

### 5.2 结构治理优先做 diff-only gate

和现有 maintainability skill 一样，结构治理也应先抓新增债务和恶化债务，而不是先试图一次性消灭全部历史问题。

也就是说，首要目标是：

- 新目录不要继续走向混用
- 旧红区不要继续接纳新结构债务
- 新的共享垃圾桶不要继续长大

### 5.3 单一职责不只发生在文件，也发生在模块

“一个文件一个主职责”只是第一层；
“一个模块一种主组织模型”“一个职责一个主入口”“一个 shared 目录一种共享边界”同样重要。

### 5.4 共享目录允许存在，但必须证明自己真共享

我们明确允许 `utils/` 和 `types/` 目录存在，但默认要满足以下条件：

- 内容被多个 feature 或多个链路共同使用
- 逻辑无状态、低耦合、边界清晰
- 没有吞入业务编排、I/O、副作用、环境读取或领域决策
- 不把单一 feature 的私有内容伪装成“共享”

如果一个文件只服务单一 feature，它默认不应放进根级 shared 目录。

### 5.5 替代必须伴随退役

任何新结构替代旧结构时，必须同时回答：

- 旧结构是否立即删除
- 若不删除，临时保留到何时
- 谁负责退役
- 退出条件是什么

不允许无限期的“先都留着”。

### 5.6 红区治理要能落到目录级别

和文件红区类似，目录结构一旦进入高风险状态，就应被冻结：

- 禁止继续增加 unrelated structure
- 触达时必须记录本次是否降低结构熵
- 必须给出下一步拆分缝，而不是只记“以后再说”

## 6. 允许与禁止的结构模式

### 6.1 允许：`utils/`、`types/` 的存在

以下情况允许存在根级 `utils/` 或 `types/`：

- 目录承载真正跨 feature 的共享内容
- 文件职责单一，边界稳定
- `utils` 主要是纯工具函数
- `types` 主要是稳定共享契约类型

这条允许不仅适用于 `utils/`、`types/`，也意味着治理不应变成“禁止某个目录名”的字面主义。

### 6.2 禁止：shared 目录掩盖 feature 边界坍塌

以下情况应视为结构风险：

- 单一 feature 私有内容长期放在根级 `utils/` 或 `types/`
- `utils` 文件里混入 DB、网络、`env` 读取、业务编排、副作用
- `types` 文件同时塞入 env、DB row、API view、内部状态模型等多类边界
- `support`、`common`、`helpers` 之类弱语义文件承担越来越多真实职责

治理对象不是目录名本身，而是 shared 容器是否正在成为边界失效后的收容所。

### 6.3 禁止：平行主入口

以下情况默认视为高风险：

- 两套路由注册器并存
- 两套主装配入口并存
- 旧 controller 和新 route handler 长期并存
- 新旧结构都可被维护者误认为主路径

这类问题默认应进入结构红区。

### 6.4 禁止：组织模型混用没有退出条件

一个模块可以处于迁移期，但不允许无限期混用。

允许：

- 迁移期间暂时同时存在 `layer-first` 和 `feature-first`
- 但必须有明确迁移目标、退出条件、负责人和下一步拆分缝

不允许：

- 长期无人负责的混合状态
- 新功能继续沿旧坏模式扩张

## 7. 建议新增的治理层：Module Structure Drift Guard

建议在现有 `post-edit-maintainability-guard` 之上，再补一层专门面向模块结构的守卫。暂定名：

- `module-structure-drift-guard`

它的定位不是替代现有 skill，而是补齐“文件级 maintainability 无法看见的结构漂移”。

### 7.1 它应检查什么

首批建议覆盖这些信号：

- `parallel-structure`
  - 同一职责出现多个可能的主入口
- `topology-mix`
  - 同一模块混用多种组织模型
- `root-scatter`
  - 根目录业务文件持续增长
- `shared-scope-mismatch`
  - 单一 feature 内容误入根级 shared 目录
- `junk-drawer-growth`
  - shared 目录或弱语义文件持续吸纳跨域内容
- `retirement-missing`
  - 新结构替代旧结构后，没有退役记录或退出条件
- `directory-hotspot`
  - 已认定高风险目录继续接纳结构债务

### 7.2 它应如何输出

和现有 maintainability skill 保持同一风格：

- 本次检查是否适用
- 检查了哪些目录/模块
- 是否存在阻塞项
- 是否存在警告项
- 每个结构风险的类型、路径、原因
- 每个风险的推荐收敛方向
- 每个风险的下一步拆分缝

### 7.3 它应如何判定 error 和 warning

默认阻塞项：

- 本次改动新增平行主入口
- 本次改动让根目录业务文件继续散落增长
- 本次改动新增 shared 垃圾桶行为
- 本次改动在已知红区目录继续新增 unrelated structure
- 本次引入新结构替代旧结构，但没有任何退役说明

默认警告项：

- 触达历史结构红区，但本次未继续恶化
- shared 目录已经很大，但本次未继续扩张
- 目录已混用多种模型，但本次仅局部触达

### 7.4 它不应该做什么

- 不做绝对目录美学裁判
- 不要求所有模块必须长得一样
- 不因为存在 `utils/` 或 `types/` 就直接报错
- 不试图自动完成全部架构重构

它的目标是阻断新增结构债务，并把存量坏结构经营成有边界、有日志的红区。

## 8. 目录级红区治理模型

建议把现有单文件 hotspot 机制扩展到目录级。

### 8.1 目录进入红区的条件

满足任一条件即可：

- 同一模块多次出现平行结构
- 一个目录持续混用多种组织模型
- 根目录业务散点文件明显增加
- shared 目录持续膨胀并跨域吸纳逻辑
- 团队已明确认定“继续在此目录内新增结构会显著提高后续维护成本”

### 8.2 目录红区的默认规则

- 不允许继续增加 unrelated structure
- 新能力优先落到目标结构，而不是继续往历史坏结构里堆
- 触达目录红区时，必须记录本次是否降低结构熵
- 必须写清下一步拆分缝和目标结构

### 8.3 建议的日志字段

和文件红区类似，但面向目录：

```md
## 结构红区触达与减债记录

### workers/nextclaw-provider-gateway-api/src
- 本次是否减债：否
- 说明：本次只补设计方案，未直接调整模块拓扑。
- 当前主要结构风险：路由装配入口并存、共享容器边界模糊、目录组织模型混用。
- 下一步拆分缝：先统一唯一主路由注册器，再按 auth / remote / admin / billing / openai 收敛 feature 边界。
```

## 9. 与现有 maintainability skill 的协同关系

建议把治理体系明确分层：

### Layer A: File / Function Maintainability

负责：

- 文件长度
- 函数复杂度
- 命名职责一致性
- 单文件红区冻结

现有承担者：

- ESLint
- `post-edit-maintainability-guard`
- `scripts/maintainability-hotspots.mjs`

### Layer B: Module Structure Drift

负责：

- 模块拓扑契约
- 平行结构探测
- shared 容器边界
- 替代后退役
- 目录红区冻结

建议承担者：

- 新的 `module-structure-drift-guard`
- 目录级 hotspot 数据源
- 结构治理日志模板

这两层必须互补，而不是互相替代。文件级可维护性解决“写得太大”，结构漂移治理解决“放得太乱、边界太混、路径不再可预测”。

## 10. 推荐的落地顺序

### Phase 1: 先固化概念和判定口径

输出物：

- 本方案文档
- 结构漂移信号清单
- `utils/types` 等 shared 目录的允许边界
- 目录红区日志模板

目标：

- 先统一语言，避免每次 review 都重新解释“什么叫结构漂移”

### Phase 2: 建目录级红区清单

输出物：

- 一份目录级 hotspot 数据源
- 每个目录红区的风险说明、允许新增、禁止新增、下一步拆分缝

目标：

- 先冻结最危险的结构区域，避免继续恶化

### Phase 3: 实现 diff-only 结构守卫

输出物：

- `module-structure-drift-guard`
- 输出格式与 `post-edit-maintainability-guard` 对齐

目标：

- 把“结构风险”从口头提醒变成默认闸门

### Phase 4: 纳入常规交付闭环

输出物：

- AGENTS / workflow / skill 文档联动
- 红区触达留痕要求
- 收尾阶段的最小检查命令

目标：

- 让结构治理成为和 lint、typecheck、smoke test 同级的默认收尾动作

## 11. 对当前样本的示范性结论

以 `workers/nextclaw-provider-gateway-api/src` 为例，本方案会把它识别成一个典型的结构漂移样本，原因不是“有 `utils` 或 `types`”，而是：

- 同一模块出现平行主入口
- 同一业务域被拆散在多个宽泛技术层目录和根目录散文件之间
- shared 容器开始承担过多跨域职责
- 组织模型混用且没有明确退出条件

也正因为如此，它是一个很好的镜子：提醒我们仓库不能只治理“大文件”，还要治理“模块结构如何逐步失真”。

---

## 12. 成功标准

这套治理的阶段性成功，不是“所有目录都长得整整齐齐”，而是：

- 维护者能预测改动该落到哪里
- 核心结构尽量只有一个主事实来源
- shared 目录不再悄悄吞噬 feature 边界
- 红区目录不再继续恶化
- 新增结构债务能被默认发现和记录
- 团队讨论结构问题时，能直接落到统一的判定口径和守卫输出，而不是凭感觉争论

这才是把局部问题当镜子之后，真正形成系统性治理闭环的标志。
